# 当前项目产品定义（基于 README_DEV 与代码现状）

## 1. 一句话定义

当前项目的本质，不是一个单纯的“个人博客模板”，也不是一个完整的“社区平台”，而是一个**面向技术型长文内容的自托管知识发布平台**：

- 前台负责内容展示、检索、阅读体验与部分互动。
- 后台负责内容生产、内容管理、评论审核、媒体管理与运行监控。
- 基础设施负责搜索索引、异步任务、对象存储、部署与观测。

如果要为未来开发建立统一方向，建议把这个项目定义为：

> 一个以“技术知识内容”为核心资产、以“内容发布与运营”为核心闭环、兼顾互动与可运维性的 API 驱动内容平台。

## 2. 当前主产品已经闭环的能力

以下能力是从代码中可以明确看出已经形成主链路的，应该视为当前产品定义的核心。

### 2.1 内容生产与发布

- 支持两种内容进入系统的方式：
  - 后台直接创建文章并发布/存草稿。
  - 将 `frontend/data/blog/` 下的 MDX 内容同步进入数据库。
- 文章以数据库为前台运行时的权威数据源，首页、博客列表、分类页、详情页都已走后端 API。
- 文章支持分类、标签、摘要、封面、SEO 字段、目录开关、布局类型、发布时间等元信息。
- 后台创建文章时会自动生成版本记录；MDX 同步时也会自动生成版本快照。
- 文章支持软删除、更新后清缓存、异步触发搜索索引更新。

### 2.2 富内容渲染

这不是一个只支持纯文本 Markdown 的系统，而是一个明显偏“技术内容表达”的平台。

- 支持 Markdown/MDX 渲染。
- 支持数学公式渲染，含行内公式与块级公式。
- 支持表格渲染。
- 支持代码块渲染。
- 支持 Excalidraw 嵌入。
- 支持乐谱/ABC 记谱渲染。
- 支持图表组件嵌入。
- 支持化学相关组件：
  - 2D 结构展示
  - 3D 结构展示
  - RDKit 渲染
  - 分子指纹展示

从产品角度看，这意味着当前平台已经具备“高信息密度技术文章”的表达能力，尤其适合机器人、化学、工程、计算机等知识型内容。

### 2.3 内容分发与发现

- 首页已能展示最新文章，并把博客内容作为站点主内容来源。
- 博客列表页支持：
  - API 拉取文章
  - 分类筛选
  - 关键词搜索
  - 分页
- 分类页支持按分类浏览文章。
- 文章详情页支持根据 slug 动态加载数据库文章。
- 搜索页已经形成明确产品能力：
  - 全文搜索
  - 自动补全建议
  - 热门搜索词
  - Meilisearch 为优先方案
  - PostgreSQL 为降级兜底
- 后端支持相关文章推荐接口。

### 2.4 读者互动

- 支持用户注册、登录、刷新令牌、退出登录、获取当前用户信息。
- 支持文章浏览量记录。
- 支持文章点赞/取消点赞。
- 支持文章评论创建与评论列表获取。
- 评论支持：
  - 回复层级
  - 点赞/取消点赞
  - HTML 清洗
  - 审核状态
- 前台文章页已经把统计与评论区整合进阅读界面。
- 文章页有相关推荐、目录、阅读进度条、评论抽屉等阅读辅助能力。

需要注意的是，这里的互动核心已经闭环的是“登录 + 点赞 + 评论 + 审核”这条链路。

### 2.5 后台运营能力

后台不是简单壳子，而是已经具备明确运营职责。

- 管理员统计总览：
  - 用户总数
  - 评论总数
  - 待审核评论数
  - 已通过/已拒绝评论数
- 用户管理：
  - 用户列表
  - 修改角色
  - 删除用户
  - 用户增长数据
- 评论管理：
  - 评论列表
  - 状态审核
  - 删除评论
  - 前端有批量操作界面
- 文章管理：
  - 管理文章列表
  - 新建文章
  - 更新/删除文章
  - 文章版本列表、版本详情、版本恢复、版本比较、版本删除
- 媒体管理：
  - 直接上传
  - 预签名上传
  - 上传完成登记
  - 媒体列表
  - 未使用媒体查询
  - 媒体元信息修改
  - 删除媒体

### 2.6 运行监控与运维可观测性

当前项目已经明显超出“只会发页面”的阶段，具备平台级运维能力。

- 后端提供健康检查：
  - `/health`
  - `/healthz`
  - `/livez`
  - `/readyz`
  - `/health/detailed`
- 后端提供 Prometheus 指标 `/metrics`。
- 前端后台提供监控页面：
  - 健康状态总览
  - 细项健康检查
  - Prometheus 指标解析展示
- 后端有 worker 处理异步 outbox 事件：
  - 搜索索引增量同步
  - 全量重建
  - 浏览量批处理落库

### 2.7 部署与工程化

当前项目已具备“可持续发布”的工程属性，这应当被视为产品能力的一部分。

- 双架构单仓：
  - `frontend/` Next.js
  - `backend/` Rust workspace
- 标准开发路径已经存在：
  - Windows PowerShell 启动脚本
  - Unix Shell 启动脚本
  - 专用 migrator
- 生产部署路径已经明确收敛为两条：
  - Compose
  - Kubernetes
- 生产基础设施支持：
  - PostgreSQL
  - Redis
  - Meilisearch
  - MinIO / 本地存储
  - Mailpit
  - Nginx edge
- CI/CD 已存在，不应再把项目理解为“手工部署网站”。

## 3. 当前产品的最佳高层定位

如果要“高屋建瓴式”重新定义，当前项目最合理的定位是：

### 3.1 核心定位

**技术知识发布与运营平台**

它的核心不是“社交”，也不是“泛内容门户”，而是围绕下面这条主线：

1. 内容创作
2. 内容入库
3. 内容渲染
4. 内容发现
5. 读者互动
6. 运营管理
7. 平台运维

### 3.2 核心资产

当前系统真正的核心资产不是页面，而是这些领域对象：

- Post
- Category
- Tag
- Media
- User
- Comment
- PostVersion
- SearchIndex
- ReadingProgress
- Operational Metrics

未来开发如果围绕这些对象组织，会比围绕“某个页面”开发更稳定。

### 3.3 主要用户角色

- 访客/读者：浏览、搜索、阅读、点赞、评论。
- 注册用户：参与互动、保存自己的阅读轨迹。
- 作者/管理员：创建内容、同步内容、管理文章、管理媒体、审核评论、管理用户。
- 运维者：部署系统、检查健康状态、观察指标、重建索引。

## 4. 应视为“产品辅翼”而非主线的能力

这些能力是存在的，但不建议把它们定义成当前项目的核心产品主线。

### 4.1 个人展示型页面

- `about`
- `projects`
- `music`

这些页面更像“站点附属展示面”，用于补充作者信息、作品与兴趣，不是系统主业务闭环。

### 4.2 本地/轻量分析型页面

- `analytics`
- `popular`
- `visitors`

这几个模块里有一部分依赖浏览器本地存储或前端文件持久化，适合做展示或实验，但不应当与后端主内容链路混为一谈。

### 4.3 内容实验能力

- 各类实验页、测试页、示例页
- 部分动画、3D、图表演示组件

它们证明系统“可扩展”，但不一定代表当前产品承诺。

## 5. 现阶段不建议写进“当前产品定义”的能力

仓库里有一些功能已经出现了页面或前端 service，但按代码现状看，还不适合视为当前稳定产品能力。

### 5.1 用户中心扩展能力未完全闭环

- 用户资料编辑
- 修改密码
- 头像上传
- 阅读统计面板

原因：

- 前端有调用，但后端主路由中未看到完整对应接口暴露。
- 这类能力更接近“计划中的用户中心”，不是当前稳定主线。

### 5.2 书签 / 阅读列表未闭环

- 前端存在书签按钮、阅读列表页、书签服务封装。
- 但后端主 API 未看到 `bookmarks` 路由注册。

因此它更像“前端预演中的能力”，而不是当前可定义的正式产品功能。

### 5.3 通知中心未闭环

- 前端存在评论通知订阅页、通知偏好与 PWA 推送设置。
- 但后端主 API 未看到对应通知路由注册。

这说明它属于明显的预留/半成品能力。

### 5.4 阅读中心能力部分存在，但契约仍需收敛

- 后端有阅读进度与阅读历史路由。
- 前端也有对应页面与 Hook。
- 但接口参数与返回结构存在不完全一致。

因此更准确的说法是：

- “阅读进度基础能力已存在”
- 但“完整的用户阅读中心”还不够稳定，不宜作为当前主产品承诺。

### 5.5 标签体系前后端尚未完全统一为主运行时

- 后端已有 tags API。
- 但前端 `tags` 页面仍明显依赖静态 Contentlayer/tag-data。

这意味着标签系统已经具备后端能力，但前台主展示链路尚未完全收敛到 API 驱动。

## 6. 对未来产品重构最有帮助的收敛建议

如果你的目标是“未来开发更加有条理”，我建议后续所有开发优先围绕下面这套边界推进。

### 6.1 建议确立的主产品边界

把产品明确定义为：

**一个面向技术长文与专业知识表达的内容平台，核心闭环是“内容生产 - 内容分发 - 用户互动 - 运营管理 - 平台运维”。**

### 6.2 建议保留为一级能力域的模块

- 内容域
  - 文章
  - 分类
  - 标签
  - 版本
  - MDX 同步
- 表达域
  - Markdown/MDX
  - 数学公式
  - 图表
  - 化学组件
  - 乐谱/Excalidraw 等嵌入
- 分发域
  - 首页
  - 列表
  - 分类页
  - 搜索
  - 推荐
- 互动域
  - 登录
  - 点赞
  - 评论
  - 评论审核
- 运营域
  - 后台统计
  - 用户管理
  - 评论管理
  - 媒体管理
  - 搜索重建
- 平台域
  - 健康检查
  - Metrics
  - Worker
  - 部署脚本
  - Compose / Kubernetes

### 6.3 建议降级为二级模块的内容

- Projects
- Music
- About
- Visitor map
- Local analytics
- Popular pages based on local analytics

这些模块可以继续存在，但在产品结构上应该放到“品牌/展示/实验区”，不要和主业务争抢架构优先级。

### 6.4 建议明确标记为 backlog / incubation 的模块

- 书签与阅读列表
- 通知中心
- 完整用户中心
- 完整标签前台 API 化
- 更成熟的阅读进度与阅读历史闭环

这些能力不是没价值，而是应该被诚实地标为“下一阶段产品化项”，而不是混入当前定义。

## 7. 最终结论

当前项目最适合被重新命名和重新组织为：

> **技术知识内容平台（Knowledge Publishing Platform）**

其已经成立的产品事实是：

- 它有真实的内容后台，而不只是静态站点。
- 它有 API 驱动的内容分发能力，而不只是编译期渲染。
- 它有丰富的专业内容表达能力，而不只是普通文章。
- 它有互动、审核、媒体、搜索、版本与运维能力，而不只是“能发文”。
- 它已经具备平台化雏形，接下来最重要的不是继续堆页面，而是**围绕主产品边界收敛能力**。

如果后续要继续推进，我建议所有 roadmap 都优先回答一个问题：

**这项开发，是在增强“技术知识内容平台”的主闭环，还是只是在给站点增加旁支页面？**

只要这个判断标准稳定下来，整个项目的未来开发就会清晰很多。
