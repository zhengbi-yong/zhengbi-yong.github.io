### C-1：Outbox 模式事务原子性失效

**严重程度**：Critical  
**影响范围**：数据一致性、搜索引擎索引可靠性  
**文件位置**：`backend/crates/api/src/routes/posts.rs`

**问题描述**

Outbox 模式的核心理念是保证"主数据操作"与"领域事件"在同一事务中原子化写入，从而消除分布式一致性问题。当前实现中，主事务 `tx.commit()` 在 outbox 事件写入**之前**就已经提交，导致两者解耦。

**问题代码**

```rust
// posts.rs 第 486-489 行（create_post）
tx.commit().await?;  // ← 主事务在此提交

// ← outbox 事件在主事务提交后才写入，不是原子操作！
if let Err(error) = crate::outbox::add_search_index_upsert(&state.db, &created_slug).await {
    tracing::warn!("Failed to add search index upsert to outbox: {error:#}");
}
```

同样的问题出现在：

| 操作 | 文件 | 行号 |
|------|------|------|
| `create_post` | `posts.rs` | 486-489 |
| `update_post` | `posts.rs` | 1015-1016 |
| `delete_post` | `posts.rs` | 1063-1064 |

**风险场景**

1. `tx.commit()` 成功 → 程序在执行 outbox 写入前崩溃 → 搜索引擎索引缺失
2. 主事务提交后 outbox 写入失败 → 搜索结果与数据库不一致，用户看到 404
3. 并发场景下，outbox worker 读取到未完成的主事务数据（取决于隔离级别）

**解决方案**

将 outbox 写入移入事务提交之前：

```rust
// ❌ 当前（错误）
tx.commit().await?;
crate::outbox::add_search_index_upsert(&state.db, &slug).await?;

// ✅ 修复后（正确）
// 在 commit 之前写入 outbox
let outbox_id = crate::outbox::add_search_index_upsert(&state.db, &slug).await?;
tx.commit().await?;

// 可选：outbox 写入后记录日志或指标
tracing::info!(outbox_id = outbox_id, "Outbox event written before commit");
```

**验证步骤**

```bash

grep -n "add_search_index_upsert\|add_.*outbox\|tx.commit" \
  backend/crates/api/src/routes/posts.rs

# 2. 确保所有 outbox 调用在 tx.commit() 之前
# 3. 运行集成测试验证事务回滚时 outbox 也被回滚
```

### C-2：后端缺失 HTTP 安全响应头

**严重程度**：Critical  
**影响范围**：XSS、点击劫持、MIME 嗅探、信息泄露  
**文件位置**：`backend/crates/api/src/main.rs`

**缺失的安全头**

| 响应头 | 作用 | 风险 |
|--------|------|------|
| `Content-Security-Policy` | 限制脚本/样式/连接来源 | XSS 攻击 |
| `Strict-Transport-Security` | 强制 HTTPS | 中间人攻击 |
| `X-Frame-Options` | 防止页面被嵌入 iframe | 点击劫持 |
| `X-Content-Type-Options` | 禁止 MIME 嗅探 | 恶意内容执行 |
| `Referrer-Policy` | 控制 Referer 头泄露 | 敏感 URL 泄露 |
| `Permissions-Policy` | 禁用危险浏览器特性 | 摄像头/麦克风滥用 |

**当前状态**

`main.rs` 仅配置了 CORS（通过 `tower_http::cors::CorsLayer`），其他安全头完全缺失。

**解决方案**

使用 `tower-http` 提供的 `SecureService` 中间件（推荐），或在 Router 层统一注入响应头。

**方案 A：使用 SecureService（推荐，5 行代码）**

```rust
// main.rs
use tower_http::secure_service::SecureService;
use tower_http::csp::Csp;
use tower_http::hsts::Hsts;
use tower_http::x_frame_options::XFrameOptions;
use tower_http::x_content_type_options::XContentTypeOptions;
use tower_http::referrer_policy::ReferrerPolicy;

// 配置安全头
let secure_service = SecureService::new(
    Csp::new()
        .default_src(tower_http::csp::ChildSrc::None)
        .script_src(tower_http::csp::ScriptSrc::Self)
        .style_src(tower_http::csp::StyleSrc::Self)
        .img_src(tower_http::csp::ImgSrc::Self)
        .connect_src(tower_http::csp::ConnectSrc::Self),
    Hsts::new().max_age_secs(31536000),  // 1 year
    XFrameOptions::new("DENY"),
    XContentTypeOptions::new(),
    ReferrerPolicy::new("strict-origin-when-cross-origin"),
);

let app = Router::new()
    .route("/", get(root))
    .layer(secure_service)
    .layer(cors_layer);
```

**方案 B：手动注入中间件（更细粒度）**

```rust
// 在 main.rs 中添加 TraceLayer 之后插入
use axum::{response::Response, middleware::Next};
use axum_extra::headers::{HeaderName, HeaderValue};
use http::{header, HeaderMap};

// 自定义安全头中间件
async fn security_headers(mut response: Response, next: Next) -> Response {
    let headers = response.headers();
    headers.insert(header::X_FRAME_OPTIONS, HeaderValue::from_static("DENY"));
    headers.insert(header::X_CONTENT_TYPE_OPTIONS, HeaderValue::from_static("nosniff"));
    headers.insert(header::X_XSS_PROTECTION, HeaderValue::from_static("1; mode=block"));
    headers.insert(
        header::REFERRER_POLICY,
        HeaderValue::from_static("strict-origin-when-cross-origin"),
    );
    headers.insert(
        header::STRICT_TRANSPORT_SECURITY,
        HeaderValue::from_static("max-age=31536000; includeSubDomains"),
    );
    // CSP 需要根据实际资源调整
    headers.insert(
        header::CONTENT_SECURITY_POLICY,
        HeaderValue::from_static(
            "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self' data:;"
        ),
    );
    next.run(response).await
}
```

```bash

  "Content-Security-Policy|Strict-Transport-Security|X-Frame-Options|X-Content-Type-Options|Referrer-Policy"
```

### C-3：前端 `@react-three/fiber` 未代码分割

**严重程度**：Critical  
**影响范围**：首屏加载性能、LCP 指标  
**文件位置**：`frontend/src/components/home/HeroSection.tsx`

**问题描述**

`HeroSection.tsx` 在组件顶层直接 `import { Canvas } from '@react-three/fiber'`，导致整个 Three.js 生态（包括 WebGL、矩阵运算、shader 编译等）被打入主 JS bundle，即使 `ParticleScene` 本身做了 dynamic import。

**问题代码**

```tsx
// HeroSection.tsx 第 5 行
import { Canvas } from '@react-three/fiber'  // ← 顶层导入，整个 three 生态进入 bundle

// 第 16-19 行
const ParticleScene = dynamic(() => import('./ParticleBackground'), { ssr: false });

// 第 55-65 行
return (
    <div className="relative h-screen w-full">
        <Canvas
            camera={{ position: [0, 0, 5], fov: 75 }}
            dpr={[1, 2]}
            gl={{ antialias: true, alpha: true }}
        >
            <ParticleScene />
        </Canvas>
    </div>
);
```

`ParticleBackground.tsx` 同样问题：

```tsx
// ParticleBackground.tsx 第 4 行
import { useFrame } from '@react-three/fiber'  // ← 直接导入 hook
```

**影响估算**

Three.js + @react-three/fiber 完整 bundle 约 **600KB-1MB**（未压缩）。这意味着用户首次访问，即使从不与 3D 场景交互，也要下载这些资源。

**解决方案**

将 Three.js 相关组件完全动态导入，并提取为独立的 chunk：

```tsx
// HeroSection.tsx — 修复后
import dynamic from 'next/dynamic';

// ✅ 正确：Three.js 相关组件全部 dynamic import
const ParticleCanvas = dynamic(
    () => import('./ParticleCanvas'),  // 封装了 Canvas + ParticleScene
    {
        ssr: false,
        loading: () => (
            <div className="absolute inset-0 bg-gradient-to-br from-slate-900 to-slate-800" />
        ),
    }
);

// 删除顶层 import { Canvas } from '@react-three/fiber'
// 删除 import { useFrame } from '@react-three/fiber'

export default function HeroSection() {
    return (
        <div className="relative h-screen w-full">
            <ParticleCanvas />
            {/* 其余 JSX（文字、按钮等）保持不变 */}
        </div>
    );
}
```

```tsx
// ParticleCanvas.tsx — 新建，封装 Canvas + Scene
'use client';

import { Canvas } from '@react-three/fiber';
import { ParticleScene } from './ParticleBackground';

export default function ParticleCanvas() {
    return (
        <Canvas
            camera={{ position: [0, 0, 5], fov: 75 }}
            dpr={[1, 2]}
            gl={{ antialias: true, alpha: true }}
        >
            <ParticleScene />
        </Canvas>
    );
}
```

### H-1：前端管理后台裸 `<img>` 标签无优化

**严重程度**：High  
**影响范围**：性能、安全（图片尺寸不可控）  
**文件位置**：`frontend/src/app/admin/media/page.tsx` 第 214、314 行

**问题代码**

```tsx
// admin/media/page.tsx:214
<img
    src={mediaUrl}
    alt={item.name}
    className="w-full h-48 object-cover rounded-lg"
/>
```

**问题影响**

1. 无 Next/Image 自动优化（WebP 转换、尺寸调整、模糊占位）
2. 无 lazy loading，首屏加载时直接请求所有图片
3. 无 `srcset`，无法适配不同设备
4. 管理员上传大尺寸图片时直接塞入 DOM

**解决方案**

使用 Next/Image 替换，并配置适当的 `sizes` 属性：

```tsx
import Image from 'next/image';

// 在 admin/media/page.tsx 中
<Image
    src={mediaUrl}
    alt={item.name}
    width={800}         // 或从 API 获取实际尺寸
    height={400}
    className="w-full h-48 object-cover rounded-lg"
    loading="lazy"      // 懒加载
    sizes="(max-width: 768px) 100vw, (max-width: 1200px) 50vw, 33vw"
/>
```

**注意**：需要确保 `next.config.ts` 中配置了图片域名：

```typescript
// next.config.ts
images: {
    remotePatterns: [
        {
            protocol: 'https',
            hostname: 'your-cdn.domain.com',
            pathname: '/media/**',
        },
    ],
},
```

```rust
once_cell::sync::Lazy::new(|| {

    ).unwrap()
});

fn validate_email(email: &str) -> bool {
    EMAIL_REGEX.is_match(email)
}

// 在 register 函数中
if !validate_email(&payload.email) {
    return Err(AppError::InvalidInput);
}
```

### H-3：文件上传图片尺寸验证是空实现

**严重程度**：High  
**影响范围**：安全（DoS 风险、图片尺寸攻击）  
**文件位置**：`backend/crates/api/src/routes/media.rs` 第 784-786 行

**问题代码**

```rust
fn get_image_dimensions(_data: &[u8]) -> Result<(Option<i32>, Option<i32>), AppError> {
    Ok((None, None))  // ← 空实现，始终返回 None
}
```

**风险**

- 用户可上传畸形/超大图片消耗存储和带宽
- 图片被用于布局攻击（极端宽高比图片导致 UI 错乱）
- 无法过滤"图片宽度 > 10000px"这类可能导致服务端处理问题的资源

**解决方案**

使用 `image` crate 实现真实尺寸检测：

```rust
// 添加到 Cargo.toml
// image = "0.25"

fn get_image_dimensions(data: &[u8]) -> Result<(Option<i32>, Option<i32>), AppError> {
    use image::GenericImageView;

    let img = image::load_from_memory(data)
        .map_err(|e| AppError::BadRequest(format!("Invalid image: {}", e)))?;

    let (width, height) = img.dimensions();
    Ok((Some(width as i32), Some(height as i32)))
}

// 在 update_media 或上传处理中调用
fn validate_image_dimensions(data: &[u8]) -> Result<(), AppError> {
    const MAX_WIDTH: i32 = 8192;
    const MAX_HEIGHT: i32 = 8192;
    const MIN_DIMENSION: i32 = 16;

    let (w, h) = get_image_dimensions(data)?
        .ok_or_else(|| AppError::BadRequest("Could not determine image dimensions".into()))?;

    if w > MAX_WIDTH || h > MAX_HEIGHT {
        return Err(AppError::BadRequest(format!(
            "Image too large: {}x{} (max: {}x{})",
            w, h, MAX_WIDTH, MAX_HEIGHT
        )));
    }
    if w < MIN_DIMENSION || h < MIN_DIMENSION {
        return Err(AppError::BadRequest(format!(
            "Image too small: {}x{} (min: {}x{})",
            w, h, MIN_DIMENSION, MIN_DIMENSION
        )));
    }
    Ok(())
}
```

```rust
use crate::middleware::auth::AdminAuth;

let app = Router::new()
    .route(
        "/sync/mdx/public",
```
