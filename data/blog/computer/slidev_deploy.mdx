## 引言

Slidev 是一个基于 Web 的演示文稿框架，它使用 Markdown 编写幻灯片，支持 Vue 组件、动画效果、演讲者注释等强大功能。在开发阶段，Slidev 作为 Web 服务器运行，提供了实时预览和编辑功能。然而，当我们需要将演示文稿分享给他人、部署到生产环境，或者需要离线访问时，就需要将 Slidev 项目构建为静态文件并进行部署。

与传统的 PowerPoint 或 Keynote 不同，Slidev 构建后的演示文稿仍然保持交互性，支持动画、代码高亮、LaTeX 数学公式等特性。这使得 Slidev 不仅适合技术分享、产品演示，也适合在线教育、文档展示等场景。

本文将深入探讨 Slidev 的构建与部署流程，从基础的 SPA（单页应用）构建开始，逐步介绍 PDF 导出、远程资源打包、多平台托管方案，以及 Docker 容器化部署。无论您是第一次部署 Slidev 项目，还是希望优化现有的部署流程，都能从本文中获得全面的指导和实践建议。

## 第一部分：构建基础

### Slidev 构建概述

Slidev 基于 Vite 构建系统，可以将 Markdown 编写的幻灯片转换为静态的 Single-page Application（SPA）。构建后的文件是完全静态的，可以部署到任何静态文件托管服务，无需服务器端支持。

**构建的核心优势**

1. **完全静态化**：构建后的文件不依赖 Node.js 运行时，可以在任何静态托管服务上运行
2. **保持交互性**：构建后的幻灯片仍然支持动画、代码高亮、LaTeX 渲染等交互功能
3. **性能优化**：Vite 构建过程会自动进行代码分割、资源优化、压缩等性能优化
4. **SEO 友好**：虽然 SPA 的 SEO 能力有限，但可以通过预渲染等技术提升搜索引擎可见性

### 基础构建命令

最简单的构建命令是：

```bash
slidev build
```

这个命令会：

- 读取项目根目录的 `slides.md` 文件（或通过配置指定的入口文件）
- 使用 Vite 进行构建和优化
- 将构建产物输出到 `dist` 目录

**构建后的目录结构**

```
dist/
├── index.html          # 主 HTML 文件
├── assets/            # 静态资源（JS、CSS、图片等）
│   ├── index-[hash].js
│   ├── index-[hash].css
│   └── ...
└── [其他静态资源]
```

**测试构建结果**

构建完成后，可以使用以下命令在本地预览构建结果：

```bash
# 使用 Vite 预览
npx vite preview

# 或使用其他静态服务器
npx serve dist
npx http-server dist
python -m http.server -d dist
```

### 自定义输出目录

默认情况下，构建产物会输出到 `dist` 目录。如果需要自定义输出目录，可以使用 `--out` 选项：

```bash
slidev build --out my-build-folder
```

这个选项在以下场景特别有用：

- 需要同时构建多个版本的演示文稿
- 希望将构建产物输出到特定的部署目录
- 需要与 CI/CD 流程集成

**示例：多环境构建**

```bash
# 开发版本
slidev build --out dist-dev

# 生产版本
slidev build --out dist-prod

# 演示版本（移除演讲者注释）
slidev build --out dist-demo --without-notes
```

### Base Path 配置

当需要将幻灯片部署到子路径（sub-route）时，必须配置 `--base` 选项。这个选项告诉 Vite 构建系统应用的基础路径，确保资源路径正确解析。

**Base Path 规则**

- Base path 必须以斜杠 `/` 开头和结尾
- Base path 应该与部署路径完全匹配
- 如果不指定 base path，默认使用 `/`（根路径）

**示例：部署到子路径**

假设您要将幻灯片部署到 `https://example.com/talks/my-cool-talk/`，构建命令应该是：

```bash
slidev build --base /talks/my-cool-talk/
```

**Base Path 的工作原理**

当设置了 base path 后，Vite 会：

1. 将所有资源路径（JS、CSS、图片等）转换为相对于 base path 的路径
2. 配置路由系统使用 base path 作为前缀
3. 确保在子路径下所有资源都能正确加载

**常见场景示例**

```bash
# GitHub Pages：用户名/仓库名
slidev build --base /my-repo/

# Netlify：部署到子目录
slidev build --base /presentations/2024/

# 自定义域名子路径
slidev build --base /slides/tech-talk/
```

**验证 Base Path**

构建完成后，检查 `dist/index.html` 文件中的资源路径，应该都包含 base path 前缀：

```html
<!-- 正确 -->
<script type="module" src="/talks/my-cool-talk/assets/index-abc123.js"></script>

<!-- 错误（缺少 base path） -->
<script type="module" src="/assets/index-abc123.js"></script>
```

## 第二部分：高级构建选项

### 移除演讲者注释

演讲者注释（Speaker Notes）是 Slidev 的一个重要功能，允许演讲者在演示时查看额外的提示信息。然而，在公开分享的演示文稿中，这些注释可能包含敏感信息或不适合公开的内容。

**使用 --without-notes 选项**

```bash
slidev build --without-notes
```

这个选项会：

- 完全移除所有演讲者注释
- 移除注释相关的 UI 组件
- 减小构建产物的体积
- 确保公开版本不包含内部信息

**何时使用**

- 公开发布的演示文稿
- 需要保护内部信息的商业演示
- 希望减小文件大小的场景
- 面向外部观众的演示

**注意事项**

使用 `--without-notes` 后，构建版本将无法查看或使用演讲者注释功能。如果需要保留注释功能，应该使用不同的构建配置或部署方案。

### 生成 PDF

Slidev 支持在构建时自动生成 PDF 版本，这对于需要离线访问、打印或存档的场景非常有用。

**启用 PDF 生成**

```bash
slidev build --pdf
```

构建过程会：

1. 正常构建 SPA 版本
2. 使用 Playwright 或 Puppeteer 渲染每一页幻灯片
3. 将渲染结果合并为 PDF 文件
4. 将 PDF 文件输出到构建目录

**PDF 生成配置**

可以在 `slides.md` 的 frontmatter 中配置 PDF 选项：

```yaml
---
# PDF 相关配置
pdf:
  # PDF 文件名
  filename: my-presentation.pdf
  # 页面大小
  format: A4
  # 打印背景
  printBackground: true
  # 页面边距
  margin:
    top: 10mm
    right: 10mm
    bottom: 10mm
    left: 10mm
---
```

**PDF 生成的最佳实践**

1. **测试渲染效果**：PDF 渲染可能与浏览器显示略有差异，建议先测试
2. **调整页面大小**：根据内容选择合适的页面格式（A4、Letter 等）
3. **处理动画**：PDF 是静态的，动画效果会被转换为最终状态
4. **优化图片**：使用适当分辨率的图片，避免 PDF 文件过大

**PDF 生成的限制**

- 不支持交互式元素（如可点击的链接在某些 PDF 阅读器中可能不工作）
- 动画效果会被转换为静态图像
- 某些 CSS 效果可能无法正确渲染
- 生成时间较长，特别是对于包含大量图片的幻灯片

### 打包远程资源

默认情况下，Slidev 构建时不会下载和打包远程资源（如图片、字体等）。如果幻灯片中引用了外部 URL 的资源，这些资源在构建后仍然需要从原始服务器加载。

**使用 --download 选项**

```bash
slidev build --download
```

这个选项会：

- 自动检测幻灯片中的远程资源
- 下载这些资源到本地
- 将资源打包到构建产物中
- 更新资源引用路径

**适用场景**

- 需要完全离线访问的演示文稿
- 外部资源可能失效或变更的情况
- 希望提升加载速度（使用 CDN 的资源）
- 需要确保资源可用性的生产环境

**注意事项**

1. **版权问题**：确保有权下载和使用远程资源
2. **资源大小**：大文件会增加构建时间和产物大小
3. **资源更新**：打包后的资源不会自动更新，需要重新构建
4. **CORS 限制**：某些资源可能因为 CORS 策略无法下载

**配置下载选项**

可以在配置文件中设置下载行为：

```typescript
// vite.config.ts
export default {
  slidev: {
    download: {
      // 下载超时时间（毫秒）
      timeout: 30000,
      // 并发下载数
      concurrency: 5,
      // 允许下载的文件类型
      allowedTypes: ['image', 'font'],
    },
  },
}
```

### 批量构建多个演示文稿

Slidev 支持一次构建多个演示文稿，这对于管理多个项目或需要构建多个版本的场景非常有用。

**构建多个文件**

```bash
slidev build slides1.md slides2.md slides3.md
```

每个输入文件会：

- 独立构建
- 生成独立的输出目录
- 保持各自的配置和资源

**使用 Glob 模式**

如果 shell 支持 glob 模式，可以使用通配符：

```bash
# 构建所有 .md 文件
slidev build *.md

# 构建特定目录下的文件
slidev build talks/*.md

# 递归构建
slidev build **/*.md
```

**批量构建的输出结构**

```
dist/
├── slides1/
│   ├── index.html
│   └── assets/
├── slides2/
│   ├── index.html
│   └── assets/
└── slides3/
    ├── index.html
    └── assets/
```

**批量构建的最佳实践**

1. **统一配置**：使用共享的配置文件确保一致性
2. **命名规范**：使用清晰的命名约定便于管理
3. **自动化脚本**：创建构建脚本简化批量构建流程
4. **版本控制**：为不同版本使用不同的输出目录

**示例：构建脚本**

```bash
#!/bin/bash
# build-all.sh

# 构建所有演示文稿
for file in talks/*.md; do
    name=$(basename "$file" .md)
    echo "Building $name..."
    slidev build "$file" --out "dist/$name"
done

echo "All presentations built successfully!"
```

## 第三部分：GitHub Pages 部署

### GitHub Pages 简介

GitHub Pages 是 GitHub 提供的免费静态网站托管服务，非常适合部署 Slidev 构建的演示文稿。它支持自定义域名、HTTPS、CDN 加速等功能，是个人项目和小型团队的首选方案。

**GitHub Pages 的优势**

1. **完全免费**：对于公开仓库完全免费
2. **自动部署**：通过 GitHub Actions 可以实现自动部署
3. **HTTPS 支持**：自动提供 SSL 证书
4. **全球 CDN**：GitHub 的 CDN 确保全球访问速度
5. **版本控制集成**：与 Git 工作流完美集成

### 配置 GitHub Pages

**第一步：启用 GitHub Actions**

1. 在 GitHub 仓库中，进入 `Settings` > `Pages`
2. 在 `Build and deployment` 部分，选择 `GitHub Actions`
3. 不要选择 `Deploy from a branch`，因为我们需要通过 Actions 构建

**第二步：创建部署工作流**

创建 `.github/workflows/deploy.yml` 文件：

```yaml
name: Deploy Pages

on:
  push:
    branches:
      - main

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: 'pages'
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v2
        with:
          version: 8

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - name: Install dependencies
        run: pnpm install

      - name: Build
        run: pnpm run build

      - name: Setup Pages
        uses: actions/configure-pages@v4

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

**工作流配置说明**

1. **触发条件**：当推送到 `main` 分支时触发
2. **权限设置**：需要 `pages: write` 和 `id-token: write` 权限
3. **并发控制**：使用 `concurrency` 避免并发部署冲突
4. **构建步骤**：安装依赖、构建项目、上传构建产物
5. **部署步骤**：将构建产物部署到 GitHub Pages

### 配置 Base Path

如果仓库名称不是 `username.github.io`，需要配置 base path。

**获取 Base Path**

Base path 格式为：`/repository-name/`

例如，如果仓库名是 `my-slides`，base path 就是 `/my-slides/`

**配置方法**

在 `vite.config.ts` 或构建命令中设置：

```typescript
// vite.config.ts
export default {
  base: process.env.NODE_ENV === 'production' ? '/my-slides/' : '/',
}
```

或者在构建命令中：

```yaml
- name: Build
  run: pnpm run build --base /my-slides/
```

**验证部署**

部署完成后，访问 `https://username.github.io/repository-name/` 查看部署结果。

### GitHub Pages 最佳实践

1. **使用 Actions 而非分支部署**：Actions 方式更灵活，支持构建过程
2. **配置缓存**：使用 Node.js 和 pnpm/npm 缓存加速构建
3. **环境变量**：使用 GitHub Secrets 管理敏感信息
4. **自定义域名**：在 Pages 设置中配置自定义域名
5. **404 页面**：创建自定义 404 页面提升用户体验

**示例：带环境变量的构建**

```yaml
- name: Build
  env:
    VITE_API_URL: ${{ secrets.API_URL }}
  run: pnpm run build
```

## 第四部分：Netlify 部署

### Netlify 简介

Netlify 是一个现代化的静态网站托管平台，提供持续部署、表单处理、服务器less 函数等功能。对于 Slidev 项目，Netlify 提供了简单易用的部署方案。

**Netlify 的优势**

1. **自动部署**：Git 推送自动触发部署
2. **预览部署**：每个 Pull Request 都有预览链接
3. **全球 CDN**：自动 CDN 加速
4. **HTTPS**：自动 SSL 证书
5. **表单处理**：内置表单处理功能（虽然 Slidev 通常不需要）

### 配置 Netlify

**第一步：创建 netlify.toml**

在项目根目录创建 `netlify.toml` 文件：

```toml
[build]
  command = "npm run build"
  publish = "dist"

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

**配置说明**

- `command`：构建命令，根据您的包管理器调整（npm/pnpm/yarn）
- `publish`：构建产物目录，通常是 `dist`
- `redirects`：SPA 路由重定向规则，确保所有路由都指向 `index.html`

**第二步：连接仓库**

1. 登录 Netlify 控制台
2. 点击 "Add new site" > "Import an existing project"
3. 选择 GitHub/GitLab/Bitbucket
4. 选择您的仓库
5. Netlify 会自动检测 `netlify.toml` 配置

**第三步：配置环境变量（如需要）**

如果构建需要环境变量，在 Netlify 控制台的 "Site settings" > "Environment variables" 中配置。

### Netlify 高级配置

**多分支部署**

```toml
[build]
  command = "npm run build"
  publish = "dist"

[context.production]
  command = "npm run build -- --base /"

[context.deploy-preview]
  command = "npm run build -- --base /preview/"
```

**构建优化**

```toml
[build]
  command = "npm run build"
  publish = "dist"

[build.environment]
  NODE_VERSION = "20"
  PNPM_VERSION = "8"
```

**自定义 Headers**

```toml
[[headers]]
  for = "/*"
  [headers.values]
    X-Frame-Options = "DENY"
    X-XSS-Protection = "1; mode=block"
    X-Content-Type-Options = "nosniff"
```

### Netlify 最佳实践

1. **使用 Netlify CLI**：本地测试部署配置
2. **配置构建缓存**：加速重复构建
3. **使用 Netlify Functions**：如果需要后端功能
4. **监控部署**：使用 Netlify Analytics 监控访问情况
5. **配置通知**：部署成功/失败时发送通知

**本地测试**

```bash
# 安装 Netlify CLI
npm install -g netlify-cli

# 本地测试构建
netlify build

# 本地预览
netlify dev
```

## 第五部分：Vercel 部署

### Vercel 简介

Vercel 是另一个流行的静态网站托管平台，特别适合前端项目。它由 Next.js 的创建者开发，对现代前端工具链有很好的支持。

**Vercel 的优势**

1. **极速部署**：优化的构建和部署流程
2. **自动优化**：自动图片优化、代码分割等
3. **边缘函数**：支持 Edge Functions
4. **预览部署**：每个提交都有预览链接
5. **Analytics**：内置分析功能

### 配置 Vercel

**第一步：创建 vercel.json**

在项目根目录创建 `vercel.json` 文件：

```json
{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/index.html"
    }
  ]
}
```

**配置说明**

- `buildCommand`：构建命令
- `outputDirectory`：构建产物目录
- `rewrites`：SPA 路由重写规则

**第二步：连接仓库**

1. 登录 Vercel 控制台
2. 点击 "Add New Project"
3. 导入 GitHub/GitLab/Bitbucket 仓库
4. Vercel 会自动检测配置

**第三步：配置项目设置**

在项目设置中可以配置：

- 环境变量
- 构建命令覆盖
- 输出目录覆盖
- 域名配置

### Vercel 高级配置

**多环境配置**

```json
{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/index.html"
    }
  ],
  "env": {
    "NODE_ENV": "production"
  }
}
```

**自定义 Headers**

```json
{
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        {
          "key": "X-Content-Type-Options",
          "value": "nosniff"
        },
        {
          "key": "X-Frame-Options",
          "value": "DENY"
        }
      ]
    }
  ]
}
```

**图片优化**

Vercel 自动优化图片，无需额外配置。如果使用 Vercel 的图片优化 API，可以在代码中使用：

```typescript
// 使用 Vercel 图片优化
const optimizedImage = `https://your-domain.vercel.app/_next/image?url=${encodeURIComponent(imageUrl)}&w=800&q=75`
```

### Vercel 最佳实践

1. **使用 Vercel CLI**：本地开发和测试
2. **配置环境变量**：区分开发和生产环境
3. **使用 Edge Functions**：需要时添加边缘函数
4. **监控性能**：使用 Vercel Analytics
5. **配置自定义域名**：使用自己的域名

**本地测试**


## 第六部分：Docker 容器化部署

### Docker 部署概述

虽然 Slidev 构建后是静态文件，但在某些场景下，使用 Docker 容器部署仍然有其优势：

1. **统一环境**：确保构建和运行环境一致
2. **易于扩展**：可以轻松扩展到多个实例
3. **隔离性**：容器提供良好的隔离性
4. **CI/CD 集成**：与容器化 CI/CD 流程集成

### 使用预构建镜像

tangramor 维护了一个 Slidev 的 Docker 镜像，可以直接使用：

**运行容器**

```bash
docker run -d \
  --name slidev \
  -p 3030:3030 \
  -v $(pwd):/slidev \
  -e SLIDEV_THEME=default \
  tangramor/slidev:latest
```

**参数说明**

- `-d`：后台运行
- `-p 3030:3030`：端口映射
- `-v $(pwd):/slidev`：挂载当前目录
- `-e SLIDEV_THEME=default`：设置主题

### 自定义 Dockerfile

**构建阶段 Dockerfile**

```dockerfile
# 构建阶段
FROM node:20-alpine AS builder

WORKDIR /app

# 复制依赖文件
COPY package*.json ./
COPY pnpm-lock.yaml ./

# 安装依赖
RUN npm install -g pnpm && pnpm install

# 复制源代码
COPY . .

# 构建
RUN pnpm run build

# 运行阶段
FROM nginx:alpine

# 复制构建产物
COPY --from=builder /app/dist /usr/share/nginx/html

# 复制 Nginx 配置
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
```

**Nginx 配置（nginx.conf）**

```nginx
server {
    listen 80;
    server_name localhost;
    root /usr/share/nginx/html;
    index index.html;

    # SPA 路由支持
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 静态资源缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Gzip 压缩
    gzip on;
    gzip_vary on;
    gzip_min_length 1024;
    gzip_types text/plain text/css text/xml text/javascript application/javascript application/xml+rss application/json;
}
```

**构建和运行**

```bash
# 构建镜像
docker build -t my-slidev .

# 运行容器
docker run -d -p 8080:80 my-slidev
```

### 常见问题与解决方案

**问题 1：资源 404 错误**

**症状**：构建后某些资源（图片、字体等）无法加载

**原因**：

- Base path 配置不正确
- 资源路径使用了绝对路径
- 资源未正确打包

**解决方案**：

```typescript
// vite.config.ts
export default {
  base: '/your-base-path/',
  assetsInclude: ['**/*.png', '**/*.jpg'],
}
```

**问题 2：路由刷新 404**

**症状**：直接访问子路由返回 404

**原因**：SPA 路由需要服务器重写规则

**解决方案**：

- Netlify：在 `netlify.toml` 中配置 redirects
- Vercel：在 `vercel.json` 中配置 rewrites
- Nginx：配置 try_files
- GitHub Pages：使用 404.html 重定向

**问题 3：构建产物过大**

**症状**：构建后的文件体积很大，加载缓慢

**解决方案**：

```typescript
// vite.config.ts
export default {
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ['vue', 'vue-router'],
        },
      },
    },
    chunkSizeWarningLimit: 1000,
  },
}
```

**问题 4：PDF 生成失败**

**症状**：构建 PDF 时出错或超时

**原因**：

- Playwright/Puppeteer 未正确安装
- 内存不足
- 某些内容无法渲染

**解决方案**：

```bash
# 确保 Playwright 已安装
npx playwright install

# 增加内存限制
NODE_OPTIONS="--max-old-space-size=4096" slidev build --pdf
```

**问题 5：动画在构建后不工作**

**症状**：本地开发正常，构建后动画失效

**原因**：可能是构建优化导致的问题

**解决方案**：

```typescript
// vite.config.ts
export default {
  build: {
    cssCodeSplit: false, // 保持 CSS 完整性
  },
}
```
