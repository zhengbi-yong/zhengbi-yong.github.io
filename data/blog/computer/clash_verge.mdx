## 第一部分：基础概念

### 网络代理基础

**代理服务器概念**

代理服务器（Proxy Server）是位于客户端和目标服务器之间的中间服务器，它接收客户端的请求，然后代表客户端向目标服务器发送请求，并将响应返回给客户端。代理服务器的主要作用包括：

- **转发请求**：将客户端的网络请求转发到目标服务器
- **隐藏身份**：隐藏客户端的真实IP地址，保护隐私
- **内容过滤**：可以过滤、缓存或修改传输的内容
- **访问控制**：可以控制哪些请求被允许或拒绝
- **性能优化**：通过缓存机制提升访问速度

**代理协议简介**

现代代理工具支持多种代理协议，每种协议都有其特点和适用场景：

**Shadowsocks（SS）**

Shadowsocks 是一种轻量级的代理协议，采用 SOCKS5 代理方式，通过加密和混淆技术实现流量转发。Shadowsocks 的特点包括：

- 轻量级，资源占用低
- 加密传输，安全性好
- 配置简单，易于部署
- 兼容性好，支持多种客户端

**V2Ray**

V2Ray 是一个功能强大的代理平台，支持多种传输协议和路由功能。V2Ray 的特点包括：

- 功能丰富，支持多种协议
- 路由能力强，可以实现复杂的流量分流
- 可扩展性好，支持插件系统
- 配置灵活，适合高级用户

**Trojan**

Trojan 是一种伪装成 HTTPS 流量的代理协议，通过 TLS 加密和流量伪装实现代理功能。Trojan 的特点包括：

- 流量伪装，难以被检测
- 使用标准 TLS 协议，兼容性好
- 性能优秀，延迟低
- 安全性高，加密强度大

**VLESS 和 Reality**

VLESS 是 V2Ray 的轻量级传输协议，Reality 是 VLESS 的增强版本，提供了更强的流量伪装能力。这些新协议的特点包括：

- 协议简单，性能更好
- 流量伪装能力强
- 支持最新的加密技术
- 适合对性能要求高的场景

**代理模式**

代理客户端通常支持三种基本模式：

**规则模式（Rule Mode）**

规则模式根据预设的规则来决定哪些流量走代理，哪些流量直连。这是最常用的模式，可以实现智能分流：

- 国内网站直连，访问速度快
- 国外网站走代理，突破限制
- 根据域名、IP、地理位置等规则分流
- 节省代理流量，降低延迟

**全局模式（Global Mode）**

全局模式将所有流量都通过代理转发，适用于需要完全隐藏IP地址的场景：

- 所有流量都经过代理
- 完全隐藏真实IP
- 可能影响国内网站访问速度
- 消耗更多代理流量

**直连模式（Direct Mode）**

直连模式不使用代理，所有流量直接连接，用于测试或临时禁用代理：

- 所有流量不走代理
- 用于测试网络连接
- 临时禁用代理功能
- 不消耗代理流量

### Clash 内核介绍

**Clash 与 Clash Meta**

Clash 是一个用 Go 语言编写的网络代理工具，提供了强大的规则引擎和代理管理功能。Clash 有两个主要版本：

**Clash（原版）**

- 功能稳定，兼容性好
- 支持 Shadowsocks、V2Ray、Trojan 等协议
- 规则引擎强大
- 社区活跃，文档完善

**Clash Meta（mihomo）**

Clash Meta 是 Clash 的增强版本，也被称为 mihomo，提供了更多功能和协议支持：

- 支持更多协议（VLESS、Reality、Hysteria 等）
- 性能优化，资源占用更低
- 功能增强，支持更多高级特性
- 持续更新，跟进最新技术发展

**Clash 核心功能**

Clash 内核提供了以下核心功能：

**规则引擎**

Clash 的规则引擎可以根据域名、IP地址、地理位置等条件，智能地将流量路由到不同的代理节点或直连。规则引擎支持：

- 域名规则（DOMAIN、DOMAIN-SUFFIX、DOMAIN-KEYWORD）
- IP 规则（IP-CIDR、GEOIP）
- 端口规则（PORT）
- 脚本规则（SCRIPT）
- 规则优先级和匹配顺序

**代理组管理**

Clash 支持多种代理组类型，实现灵活的代理管理：

- **Selector（选择器）**：手动选择代理节点
- **URL-Test（自动选择）**：根据延迟自动选择最优节点
- **Fallback（故障转移）**：按顺序尝试节点，失败自动切换
- **Load-Balance（负载均衡）**：在多个节点间分配流量
- **Relay（链式代理）**：通过多个代理节点转发流量

**流量统计**

Clash 可以统计每个代理节点的流量使用情况，帮助用户了解流量消耗：

- 实时流量统计
- 历史流量记录
- 流量限制和警告
- 流量重置功能

**DNS 处理**

Clash 内置 DNS 服务器，可以处理 DNS 查询，实现 DNS 分流和缓存：

- DNS 服务器配置
- DNS 分流规则
- DNS 缓存机制
- 防止 DNS 泄漏

### 代理模式

**规则模式详解**

规则模式是 Clash Verge 最常用的模式，它根据预设规则智能分流：

**工作原理**

1. 当应用程序发起网络请求时，Clash 会检查请求的目标
2. 根据规则列表，从上到下匹配规则
3. 匹配到规则后，将流量路由到指定的代理组或直连
4. 如果没有匹配到规则，使用默认规则（通常是直连或代理）

**规则类型**

**域名规则**

- `DOMAIN`：精确匹配域名
- `DOMAIN-SUFFIX`：匹配域名后缀
- `DOMAIN-KEYWORD`：匹配域名关键词
- `DOMAIN-SUFFIX,example.com,Proxy`：example.com 及其子域名走代理

**IP 规则**

- `IP-CIDR`：匹配 IP 地址段
- `GEOIP`：根据地理位置匹配
- `IP-CIDR,192.168.0.0/16,DIRECT`：内网地址直连
- `GEOIP,CN,DIRECT`：中国IP直连

**端口规则**

- `PORT`：匹配端口号
- `PORT,443,Proxy`：443端口走代理

**脚本规则**

- `SCRIPT`：使用 JavaScript 脚本自定义规则逻辑

**规则优先级**

规则按照配置文件中的顺序从上到下匹配，匹配到第一个规则后停止。因此，更具体的规则应该放在前面，更通用的规则放在后面。

**全局模式使用**

全局模式将所有流量都通过代理转发：

**适用场景**

- 需要完全隐藏IP地址
- 访问所有网站都需要代理
- 测试代理连接
- 临时需要全局代理

**注意事项**

- 国内网站访问速度可能变慢
- 消耗更多代理流量
- 可能影响某些需要直连的应用
- 建议仅在必要时使用

**直连模式应用**

直连模式不使用代理，所有流量直接连接：

**适用场景**

- 测试网络连接
- 临时禁用代理
- 访问不需要代理的网站
- 节省代理流量

**TUN 模式配置**

TUN 模式通过创建虚拟网卡，实现系统级别的全局代理：

**TUN 模式优势**

- 接管所有系统流量，包括不支持系统代理的应用
- 更彻底的流量控制
- 支持 UDP 流量代理
- 可以实现真正的全局代理

**启用 TUN 模式**

1. 打开 Clash Verge 设置
2. 找到"TUN 模式"选项
3. 启用 TUN 模式
4. 根据系统提示授予必要权限：
   - Windows：需要管理员权限
   - macOS：需要在系统偏好设置中允许
   - Linux：需要 root 权限或配置 sudo

**TUN 模式配置**

在配置文件中可以设置 TUN 模式的详细参数：

```yaml
tun:
  enable: true
  stack: system # 或 gvisor
  dns-hijack:
    - any:53
  auto-route: true
  auto-detect-interface: true
```

## 第四部分：高级配置与应用

### 规则配置

**规则编写基础**

Clash 的规则系统非常强大，可以精确控制流量路由：

**规则格式**

```
规则类型,匹配值,目标代理组
```

**规则示例**

```
DOMAIN-SUFFIX,google.com,Proxy
IP-CIDR,192.168.0.0/16,DIRECT
GEOIP,CN,DIRECT
MATCH,Proxy
```

**域名规则**

**DOMAIN（精确匹配）**

- 格式：`DOMAIN,example.com,Proxy`
- 说明：精确匹配 example.com 域名
- 示例：`DOMAIN,www.google.com,Proxy`

**DOMAIN-SUFFIX（后缀匹配）**

- 格式：`DOMAIN-SUFFIX,example.com,Proxy`
- 说明：匹配 example.com 及其所有子域名
- 示例：`DOMAIN-SUFFIX,google.com,Proxy` 会匹配 google.com、www.google.com、mail.google.com 等

**DOMAIN-KEYWORD（关键词匹配）**

- 格式：`DOMAIN-KEYWORD,google,Proxy`
- 说明：匹配包含关键词的域名
- 示例：`DOMAIN-KEYWORD,google,Proxy` 会匹配所有包含 "google" 的域名

**DOMAIN-REGEX（正则匹配）**

- 格式：`DOMAIN-REGEX,^.*\.google\.com$,Proxy`
- 说明：使用正则表达式匹配域名
- 示例：匹配所有以 .google.com 结尾的域名

**IP 规则**

**IP-CIDR（IP 段匹配）**

- 格式：`IP-CIDR,192.168.0.0/16,DIRECT`
- 说明：匹配指定 IP 段的流量
- 示例：
  - `IP-CIDR,192.168.0.0/16,DIRECT`：内网地址直连
  - `IP-CIDR,10.0.0.0/8,DIRECT`：内网地址直连
  - `IP-CIDR,172.16.0.0/12,DIRECT`：内网地址直连

**GEOIP（地理位置匹配）**

- 格式：`GEOIP,CN,DIRECT`
- 说明：根据 IP 地理位置匹配
- 常用国家代码：
  - `CN`：中国
  - `US`：美国
  - `JP`：日本
  - `KR`：韩国
  - `HK`：香港
  - `TW`：台湾
  - `SG`：新加坡

**地理位置规则**

GEOIP 规则可以根据 IP 地址的地理位置进行路由：

**使用场景**

- 国内网站直连：`GEOIP,CN,DIRECT`
- 特定国家走代理：`GEOIP,US,Proxy`
- 排除特定地区：`GEOIP,CN,DIRECT` 放在前面

**注意事项**

- GEOIP 数据库需要定期更新
- 某些 IP 可能无法准确识别地理位置
- 建议结合域名规则使用

**规则优先级**

规则的匹配顺序非常重要，Clash 按照规则列表从上到下匹配：

**优先级原则**

1. 更具体的规则应该放在前面
2. 更通用的规则放在后面
3. 匹配到规则后立即停止，不再继续匹配
4. `MATCH` 规则应该放在最后，作为默认规则

**规则顺序示例**

```yaml
rules:
  # 1. 最具体的规则：精确域名匹配
  - DOMAIN,www.google.com,Proxy

  # 2. 次具体的规则：域名后缀匹配
  - DOMAIN-SUFFIX,google.com,Proxy

  # 3. 通用规则：地理位置匹配
  - GEOIP,CN,DIRECT

  # 4. 默认规则：匹配所有
  - MATCH,Proxy
```

**规则优化建议**

- 将常用的规则放在前面，提高匹配效率
- 使用域名规则而不是 IP 规则，更精确
- 合理使用 GEOIP 规则，减少规则数量
- 定期更新规则列表，移除无效规则

### 代理组配置

**代理组类型**

Clash 支持多种代理组类型，每种类型有不同的用途：

**Selector（选择器）**

手动选择代理节点，适合需要精确控制的场景：

```yaml
proxy-groups:
  - name: '手动选择'
    type: select
    proxies:
      - '节点1'
      - '节点2'
      - '节点3'
      - DIRECT
```

**URL-Test（自动选择）**

根据延迟自动选择最优节点：

```yaml
proxy-groups:
  - name: '自动选择'
    type: url-test
    proxies:
      - '节点1'
      - '节点2'
      - '节点3'
    url: 'http://www.gstatic.com/generate_204'
    interval: 300
    tolerance: 50
```

- `url`：测试 URL
- `interval`：测试间隔（秒）
- `tolerance`：延迟容差（毫秒），只有延迟差异超过此值才切换

**Fallback（故障转移）**

按顺序尝试节点，失败自动切换：

```yaml
proxy-groups:
  - name: '故障转移'
    type: fallback
    proxies:
      - '节点1'
      - '节点2'
      - '节点3'
    url: 'http://www.gstatic.com/generate_204'
    interval: 300
```

**Load-Balance（负载均衡）**

在多个节点间分配流量：

```yaml
proxy-groups:
  - name: '负载均衡'
    type: load-balance
    proxies:
      - '节点1'
      - '节点2'
      - '节点3'
    url: 'http://www.gstatic.com/generate_204'
    interval: 300
    strategy: round-robin # 或 consistent-hashing
```

**Relay（链式代理）**

通过多个代理节点转发流量：

```yaml
proxy-groups:
  - name: '链式代理'
    type: relay
    proxies:
      - '节点1'
      - '节点2'
```

**负载均衡**

负载均衡可以在多个节点间分配流量，提高性能和可靠性：

**策略类型**

- `round-robin`：轮询方式，依次使用每个节点
- `consistent-hashing`：一致性哈希，根据目标地址选择节点

**配置示例**

```yaml
proxy-groups:
  - name: '负载均衡组'
    type: load-balance
    proxies:
      - '节点1'
      - '节点2'
      - '节点3'
    url: 'http://www.gstatic.com/generate_204'
    interval: 300
    strategy: round-robin
```

**故障转移**

故障转移可以自动切换到可用的节点：

**工作原理**

1. 按顺序尝试每个节点
2. 测试节点是否可用
3. 如果节点失败，自动切换到下一个节点
4. 定期重新测试失败的节点

**配置示例**

```yaml
proxy-groups:
  - name: '故障转移组'
    type: fallback
    proxies:
      - '主节点'
      - '备用节点1'
      - '备用节点2'
    url: 'http://www.gstatic.com/generate_204'
    interval: 300
```

**自动选择**

自动选择功能可以根据延迟自动选择最优节点：

**工作原理**

1. 定期测试所有节点的延迟
2. 选择延迟最低的节点
3. 如果延迟差异超过容差值，自动切换节点

**配置示例**

```yaml
proxy-groups:
  - name: '自动选择组'
    type: url-test
    proxies:
      - '节点1'
      - '节点2'
      - '节点3'
    url: 'http://www.gstatic.com/generate_204'
    interval: 300
    tolerance: 50
```

```bash
#!/bin/bash

hour=$(date +%H)
if [ $hour -ge 9 ] && [ $hour -lt 18 ]; then
    # 工作时间使用工作节点
    clash-verge switch-node "工作节点"
else
    # 非工作时间使用备用节点
    clash-verge switch-node "备用节点"
fi
```
